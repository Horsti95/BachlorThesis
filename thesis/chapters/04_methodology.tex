\section{Theoretical Framework}
\label{sec:theoretical}

The central engineering challenge in LOSO caching is determining when two
experimental runs are \emph{identical enough} to reuse prior results. A
traditional approach would compare configuration files by hand or rely on
naming conventions, both of which are error-prone and do not scale to
2{,}304 operations across 18 configurations. The framework introduced here
instead encodes every relevant parameter into a deterministic cryptographic
key, so correctness becomes structural rather than a matter of human
discipline. Three design concepts underpin this approach.

\subsection{Configuration Fingerprinting}

Configuration fingerprinting provides the foundation for cache key generation
and reproducibility tracking. Each experiment configuration --- comprising
model hyperparameters, feature-selection settings, random seed, code version,
and the held-out subject identifier --- is serialised to a canonical JSON
string and hashed with SHA-256, truncated to 32 hex characters (128 bits)
for filesystem compatibility:

\begin{center}
\texttt{fingerprint = sha256(canonical\_json(config))[:32]}
\end{center}

\noindent
The 32-character prefix provides ample collision resistance at the
experimental scale ($\sim$2{,}300 configurations), and the canonical
serialisation (\texttt{sort\_keys=True}, compact separators) ensures
identical output regardless of dictionary insertion order.
Table~\ref{tab:fingerprint_fields} lists the tracked fields.

The held-out subject field is the critical LOSO-specific addition: it encodes
the fold identity directly into the key, ensuring a model cached for subject
$S_i$ can never be retrieved for subject $S_j$. An example combined cache
key: \texttt{a3f2b8c1...\_SC4001.joblib}.

The fingerprint generation module is fully implemented, providing the \texttt{LOSOFingerprint} class with methods for generating fingerprints from configuration objects or dictionaries. The \texttt{generate\_cache\_key()} function provides a convenient interface that combines the fingerprint with the held-out subject ID (e.g., \texttt{a3f2b8c1...\_SC4001}) for direct use as cache file names.

\subsection{Configuration-Independent vs. Configuration-Dependent Caching}
\label{sec:storage_vs_caching}

A fundamental distinction in this framework is between \textit{configuration-independent caching} (artifacts that depend only on the dataset) and \textit{configuration-dependent caching} (artifacts that require fingerprint-based invalidation when experimental parameters change). While both avoid redundant computation, they serve different purposes:

\textbf{Configuration-Independent Cache} refers to artifacts that are computed once per dataset version and remain valid across all experimental variations. These artifacts do not depend on model hyperparameters, feature selection criteria, or cross-validation strategies. In this framework, the extracted feature matrices constitute configuration-independent caching---they are computed once per subject and reused across all experiments.

\textbf{Configuration-Dependent Cache} refers to artifacts whose validity depends on experimental configuration. These artifacts require fingerprint-based invalidation when parameters change. The LOSO model cache, which stores trained models keyed by configuration fingerprint and held-out subject, exemplifies configuration-dependent caching.

\subsubsection{Layer 1: Feature Extraction as Persistent Storage}

The feature extraction layer computes \textbf{all 149 features} for each subject across 6 EEG channels (F3, F4, C3, C4, O1, O2) and stores them in compressed NPZ format. This is classified as persistent storage because:

\begin{itemize}
    \item Features depend only on the raw EEG data and fixed preprocessing parameters
    \item Changing model hyperparameters or feature selection thresholds does not invalidate the stored features
    \item The full feature set is extracted once; downstream selection operates on this cached superset
\end{itemize}

In this implementation, the pipeline checks for \texttt{\{subject\_id\}.npz}
before extracting; if found, it is loaded directly and extraction is skipped.
This layer is fully implemented and provides measured speedups of approximately 227$\times$ for feature loading compared to re-extraction.

\subsubsection{Layer 2: LOSO Model Caching}
\label{sec:loso_caching}

The LOSO model cache stores one trained model per fingerprint--subject pair.
The complete system---fingerprint generation, cache storage, retrieval logic,
and automatic invalidation---is validated through verification testing
(Section~\ref{sec:verification}) and the full-scale 18-configuration experiment
(Section~\ref{sec:results_caching}).

\textbf{Full-Scale Results:} The caching system achieved 100\% cache hit rate
across all 18 configurations and 128 LOSO folds (2{,}304 total operations).
Measured speedups for XGBoost ranged from 37.5$\times$ to 202$\times$ (median
54$\times$) with cache sizes of $\sim$185~MB per 9-configuration sweep
(128 models per configuration, $\sim$1.4~MB per fold). Random Forest achieved
11--23.5$\times$ speedup (median 11.6$\times$) but required $\sim$18.5~GB per
configuration ($\sim$145~MB per fold).

\textbf{Why LOSO-Level Caching, Not Just Model Caching?}

A critical design decision is caching at the LOSO fold level rather than caching a single global model. This distinction is essential:

\begin{enumerate}
    \item \textbf{Data Leakage Prevention:} Each LOSO fold has a unique fingerprint that includes the \texttt{held\_out\_subject} field. A model trained with Subject~A held out cannot be retrieved when Subject~B is the test subject---the fingerprints differ (as demonstrated by the implemented \texttt{LOSOFingerprint} class), forcing a cache miss and retraining.
    
    \item \textbf{Scientific Validity:} In LOSO cross-validation, each fold produces a different model because the training data differs. There is no single ``global model''---there are $N$ distinct models, one per subject held out; reusing any one for a different held-out subject would apply a model trained on different data.
    
    \item \textbf{Incremental Experimentation:} When adding new subjects or modifying experimental parameters, only affected folds require retraining. Cached models for unchanged configurations remain valid and are reused automatically.
\end{enumerate}

\subsection{Dataset Selection Justification}

BOAS was selected for this benchmark for four complementary reasons.

\textbf{Scale for LOSO.} With 128 subjects, BOAS yields 128 LOSO folds---sufficient
to measure cache hit rates, speedup distributions, and scaling behaviour with
statistical reliability. Smaller datasets would compress the scaling experiments
and reduce confidence in the median speedup estimates.

\textbf{Label quality.} BOAS provides a consensus label derived from three independent
scorers and a fourth senior arbitrator~\cite{esparza2026automatic}, reducing the
impact of inter-rater variability on model training. The dataset creators report
87.08\% human-AI agreement on PSG data, providing a natural performance ceiling
for comparison.

\textbf{Continuity.} The BOAS dataset was used in the prior course project from which
this thesis evolved. Retaining the same dataset preserves the experimental conditions
under which the caching bottleneck was first identified, enabling direct comparison
between the uncached and cached workflows that motivated this research.

\textbf{Public availability.} BOAS is publicly accessible~\cite{boas_dataset}, allowing
external researchers to reproduce the benchmarks on identical data without requiring
institutional access or dataset licensing agreements.

\subsection{Class Distribution and LOSO Justification}

Across all 128 subjects the 119,759 labelled epochs break down as:
Wake~16.0\%, N1~3.7\%, N2~60.3\%, N3~4.4\%, REM~15.7\%
(Figure~\ref{fig:sleep_distribution}). N2 dominates while N1 is nearly
16$\times$ less frequent, motivating macro-averaged F1 as the primary metric.

The aggregate distribution masks substantial inter-subject variability.
Figure~\ref{fig:subject_variation} shows four subjects selected as extreme
cases from the full 128-subject pool: sub-114 spends 80\% of recorded time
in N2 with no N3 epochs at all; sub-34 is awake for over half of the
recording (51.5\% Wake); sub-2 has an unusually high N3 fraction (22.6\%);
and sub-119 has the highest REM fraction in the dataset (25.2\%).
A model trained on sub-114 would never observe N3 training examples
and would generalise poorly to sub-2; conversely, a model trained
predominantly on Wake-heavy subjects would be poorly calibrated for
subjects with near-zero Wake epochs.
This variability is precisely why Leave-One-Subject-Out cross-validation
is required: each subject must serve as the held-out test set at least
once, so that evaluation is never contaminated by the subject-specific
distribution the model was trained on.

\subsection{Data Validation and Integrity Verification}
\label{sec:data_validation}

Prior to feature extraction, each recording undergoes an automated 12-point
validation protocol covering annotation integrity (sufficient epochs, all
5 stages present, duration $\geq$2~hours, filter rate $<$30\%, temporal
continuity), structural consistency (correct channel count and sampling rate),
EDF-annotation alignment (onset bounds, sample bounds), and signal quality
(no NaN/$\pm\infty$ values, no flat channels, amplitude $<10^6~\mu$V).

All 128 BOAS subjects passed all checks. The dataset exhibits minimal
artifact contamination ($<$0.1\% epoch exclusion rate). The dataset
creators report a human-AI agreement of 87.08\% for PSG
data~\cite{esparza2026automatic}, which serves as a performance benchmark
in this thesis.




\section{Preprocessing Pipeline}
\label{sec:preprocessing}

The preprocessing pipeline transforms raw EDF recordings into clean, epoched signals suitable for feature extraction.

\subsection{Signal Filtering}

The following signal conditioning is applied sequentially to each EEG channel:

\begin{enumerate}
\item \textbf{Bandpass Filter:} A 5th-order Butterworth filter with cutoff frequencies of 0.5--40 Hz removes DC drift and high-frequency noise while preserving sleep-relevant frequency bands.
\item \textbf{Notch Filter:} A notch filter at 50 Hz (Q=30) removes power line interference.
\item \textbf{Downsampling:} 256 Hz $\rightarrow$ 128 Hz using decimation with built-in anti-aliasing low-pass filter (prevents aliasing artifacts).
\end{enumerate}

All preprocessing parameters are included in the cache fingerprint, ensuring automatic invalidation when parameters change.

\subsection{Epoch Creation}

Continuous signals are segmented into 30-second epochs aligned with hypnogram annotations. Each epoch contains 3,840 samples (30 seconds $\times$ 128 Hz) across 6 channels, resulting in an array of shape $(n_\text{epochs}, 3840, 6)$ per subject.

\subsection{Quality Control}

The pipeline includes an optional per-epoch rejection step (threshold
$\pm$1{,}000~$\mu$V; flags NaN/infinite values and flat channels), but this
is disabled in the main benchmark pipeline (\texttt{validate\_quality=False}).
Recording-level quality screening is performed by the 12-point subject
validation described in Section~\ref{sec:data_validation}; the BOAS dataset
shows minimal artefact contamination ($<$0.1\% of epochs), making per-epoch
rejection unnecessary for this study.

\subsection{Feature Engineering Strategy and Caching Prioritization}

While Satapathy et al.\ (2024) demonstrate competitive performance using 63 multi-modal features~\cite{satapathy2024multimodal}, this thesis extracts an expanded set of 149 EEG-specific features. The larger set serves the caching evaluation: it increases extraction cost (providing a stronger stress test for Layer~1) and enables systematic comparison of feature-count cut-offs as a configuration parameter. Because all 149 features are stored in the persistent Layer~1 cache, downstream experiments select any sub-set (top-30, top-50, or all 149) with near-zero latency, without re-extracting from raw signals.

\subsection{Feature Selection Strategy}

After extracting all 149 features, optional feature selection is applied using a two-stage pipeline:

\textbf{Stage 1: Correlation-Based Filtering}
\begin{itemize}
\item Removes highly correlated features using Pearson correlation
\item Threshold options: 0.75, 0.90, or None (no filtering)
\item Prevents multicollinearity in downstream models
\end{itemize}

\textbf{Stage 2: ANOVA F-statistic Selection}
\begin{itemize}
\item Ranks features by class-discriminative power using F-statistic (\texttt{f\_classif})
\item Top-K options: 30, 50, or None (retain all 149 features)
\item ANOVA selected over Mutual Information based on a pilot benchmark (10 subjects, 5-fold CV): ANOVA was 174$\times$ faster \textit{and} 3.68 percentage points more accurate, giving strict dominance with no trade-off (Table~\ref{tab:anova_vs_mi}).
\end{itemize}

\textbf{Methodological note on feature-selection scope.}
Feature selection is performed \textbf{globally} (\texttt{scope=global}): the
ANOVA selector is fitted once on all 128 subjects and the resulting feature
set is reused across all LOSO folds, rather than being refitted independently
inside each fold (\texttt{scope=per\_fold}).

Per-fold refitting is the strict best practice in MLOps because it eliminates
information leakage from the held-out subject's labels into the feature
ranking. The global variant is used here for computational efficiency in the
caching study, with the supervisor's prior approval. Performance numbers in
this thesis should therefore be interpreted as \textit{upper-bound estimates}
of generalisation accuracy rather than fully unbiased scores: the $1/128
\approx 0.8\%$ contribution of the held-out subject's labels to the global
ranking produces a minor optimistic bias. An empirical benchmark
(Table~\ref{tab:global_vs_fold}) measured this effect at $\leq 1.1\%$
accuracy difference with no consistent direction, confirming the bias is
negligible in practice.

The framework supports \texttt{scope=per\_fold} and the same 18-configuration
experimental grid can be re-run under that setting if a future replication
requires strict isolation (Section~\ref{sec:limitations}).

\subsection{Feature Extraction Optimization}

To enable efficient processing of the BOAS dataset (128 subjects,
$\sim$950--1{,}000 epochs per subject), the extraction pipeline uses
process-based parallelization via joblib to bypass Python's GIL for
CPU-bound NumPy/SciPy operations. Complexity features are accelerated using
the antropy library where applicable---specifically DFA (\texttt{ant.detrended\_fluctuation})
and Hjorth parameters (\texttt{ant.hjorth\_params}); the Hurst exponent retains a
standard R/S-analysis implementation to avoid duplicating the DFA computation.
Each subject's full feature set is computed once and stored as a compressed NumPy
archive; downstream experiments filter to the required channel count and top-K
selection at runtime without re-extracting from raw signals.

On the benchmark desktop, cold extraction of all 128 subjects takes
approximately 53~minutes (parallelised extraction + cache write); a warm
run reading from the Layer~1 cache completes in under one minute
($227\times$ speedup).

\subsection{Fingerprint Generation}
\label{sec:fingerprint_generation}

Cache key generation follows a five-step process:

\begin{enumerate}
\item \textbf{Configuration assembly.} All tracked parameters are combined
      into a single dictionary: \texttt{random\_seed}, \texttt{code\_version},
      \texttt{model\_config} (algorithm name and hyperparameters),
      \texttt{feature\_config} (base feature count, correlation threshold,
      top-K selection, and a 16-character SHA-256 hash of the sorted selected
      feature names), and \texttt{held\_out\_subject}. The feature list is
      represented as a compact hash (\texttt{features\_hash}) rather than the
      full name list to keep the fingerprint string short.

\item \textbf{Canonical serialization.} The dictionary is serialized to JSON
      with \texttt{sort\_keys=True} and compact separators \texttt{(',',':')}
      (\texttt{json.dumps}), producing a deterministic string regardless of
      dictionary insertion order.

\item \textbf{SHA-256 hashing and truncation.} The canonical JSON string is
      hashed with SHA-256 (256 bits = 64 hex characters) and then truncated to
      32 hex characters (128 bits) for filesystem compatibility. 128 bits
      provides ample collision resistance at the experimental scale
      ($\sim$2{,}300 configurations).

\item \textbf{Cache key formation.} The 32-character fingerprint is
      concatenated with the held-out subject identifier:
      \texttt{\{fingerprint\}\_\{subject\}}, e.g.\
      \texttt{a3f2b8c1...\_sub-001}.
\end{enumerate}

The implementation in \texttt{fingerprint.py} (steps 2--4):

\begin{verbatim}
canonical = json.dumps(config.to_dict(),
                       sort_keys=True, separators=(',',':'))
fingerprint = hashlib.sha256(canonical.encode()).hexdigest()[:32]
cache_key = f"{fingerprint}_{held_out_subject}"
\end{verbatim}

\textbf{Partial cache reuse across experiment sweeps.} Configurations sharing
identical hyperparameters and subject folds produce matching fingerprints,
enabling automatic reuse. For example, sweeps $\{A_1, A_2, B_1, B_2\}$ and
$\{B_1, B_2, C_1, C_2\}$ reuse cached $B_1$, $B_2$ models while training
$C_1$, $C_2$ fresh. This was verified via incremental sweep expansion
(Section~\ref{sec:verification}).

\textbf{Design note: why \texttt{features\_hash} is in the fingerprint.}
During development, a cache-collision bug occurred: two fingerprints with
identical selection parameters (same threshold, same top-K) but different
resulting feature subsets matched on disk, causing a model trained on 37
features to be loaded for a configuration that had selected 33. The fix was
to include a 16-character SHA-256 hash of the sorted selected feature names
(\texttt{features\_hash}) in the fingerprint. A change in the actual feature
set now produces a different key even when threshold and top-K are unchanged.

\textbf{Version control safety.} The \texttt{code\_version} parameter
(set to the current semantic version or Git commit hash) ensures that
implementation changes automatically invalidate the cache. If an implementation
migration introduces a bug or modifies model behaviour, the resulting fingerprint
differs from previous versions, preventing retrieval of stale cached
models.

\subsection{Cache Storage Structure}

The two-layer directory layout reflects the key/no-key distinction:
Layer~1 filenames carry no fingerprint (features are configuration-independent);
Layer~2 filenames embed the 32-character fingerprint so any parameter or
subject change forces a cache miss and prevents accidental reuse.

Layer~1 files follow the fingerprint-free pattern
\texttt{features/sub-N\_full.npz} (always valid, never invalidated).
Layer~2 files embed the fingerprint in their filename:
\texttt{models/\{fp32\}\_sub-N.joblib}, so any parameter or subject
change forces a cache miss. A companion \texttt{models/registry.json}
maps each fingerprint to its full configuration metadata.

\subsection{Cache Hit/Miss Logic}

For each LOSO fold, the system generates the configuration fingerprint and
checks whether the corresponding joblib file exists on disk. On a cache hit,
the model is loaded (XGBoost: $\sim$58~ms; RF: $\sim$2.5~s). On a cache miss,
the model is trained (XGBoost: $\sim$29~s; RF: $\sim$248~s) and saved for
future reuse. This single-tier disk-based approach suits the batch-oriented
workflow, where each model is accessed exactly once per run.

An in-memory LRU tier was considered but not implemented: because each model
is accessed once per run, an in-memory layer would never produce a second hit.
Two-tier RAM+disk caching is discussed as a future extension in Chapter~5.

\subsection{Cache Invalidation}

Two invalidation strategies handle data or code changes:

\begin{enumerate}
\item \textbf{Version bumping.} Incrementing the \texttt{code\_version} string
in the configuration changes the fingerprint, automatically invalidating all
cached entries without deleting files. The \texttt{code\_version} field is
currently set to a fixed semantic string (\texttt{"1.0.0"}); in a CI/CD
deployment it would be populated automatically from the Git commit hash
(e.g.\ \texttt{git rev-parse HEAD}), so that any code change implicitly
invalidates dependent caches without manual intervention. Manual bumping is
sufficient for the controlled scope of this thesis, where the codebase is
frozen for the experimental run, but a production deployment should automate
this derivation.

\item \textbf{Cache purging.} For major data or model changes, the cache
directory is deleted entirely, triggering full recomputation on the next run.
\end{enumerate}

Both strategies are available; the benchmark uses version bumping for
controlled invalidation tests and purging for full cold-start baselines.

Absolute timing values are hardware-dependent. Cross-machine verification

workstation, Table~\ref{tab:hardware_env}) confirms two distinct portability
properties:

\begin{itemize}
\item \textbf{VIABLE/NOT\_VIABLE verdicts are hardware-portable.} Both
      machines agree: XGBoost is VIABLE, Random Forest is BORDERLINE. This
      holds because viability is determined by serialization size relative to
      load time, not by absolute timing values.
\item \textbf{Speedup ratios are \textit{not} hardware-portable.} The RTX~5090
      machine trains $3.4\times$ faster than the benchmark desktop (cold run:
      1{,}124~s vs.\ 3{,}773~s for XGBoost), but warm cache load times are
      nearly identical ($\sim$20~s on both). Since training is CPU-bound and
      cache loading is I/O-bound, a faster CPU compresses the cold time without
      proportionally reducing the warm time, yielding a lower speedup ratio:
      XGBoost 54$\times$ on the RTX~5090 vs.\ 188$\times$ on the benchmark
      desktop (same configuration, corr=None, k=None).
      The corollary is practically relevant: \textit{caching is most
      beneficial on slower hardware.} A researcher on a standard laptop who
      trains $3\times$ slower than the benchmark desktop will observe speedup
      ratios approximately $3\times$ higher, while cache load time (I/O-bound)
      remains constant. The framework therefore delivers its largest gains
      precisely where compute resources are most constrained.
\end{itemize}

All benchmark timings were measured with empty cache directories as the
controlled condition, ensuring results are not inflated by SSD degradation
under high fill levels.

OS page cache effect was measured explicitly on the benchmark desktop by
comparing warm runs with and without a page cache flush. For both XGBoost
($\approx$185~MB cache) and Random Forest ($\approx$18.5~GB cache), the
flush made no meaningful difference: warm times changed by less than 10\%
(XGBoost: 20.1~s no-flush vs.\ 18.8~s flushed; Random Forest: 356.9~s
vs.\ 323.6~s). Warm-run timings therefore reflect SSD read performance rather
than OS-level page caching.

\subsection{Cross-Validation and Evaluation Scenarios}

Each of the 18 configurations is evaluated under 128-fold LOSO

(2)~identical re-run (all hits, measures maximum speedup); (3)~changed
preprocessing (invalidates both layers); (4)~changed feature selection
(Layer~1 hit, Layer~2 miss per changed config); (5)~new subjects added
(existing subjects hit, new subjects miss).

\section{Evaluation Metrics}
\label{sec:evaluation_plan}

The experimental evaluation used three categories of metrics, corresponding to
the primary caching contribution, the secondary classification validation, and
the reproducibility requirements.

\subsection{Primary Metrics: Caching Efficiency}

The primary thesis contribution was evaluated through caching metrics:

\begin{itemize}
\item \textbf{Cache Hit Rate:} $\frac{\text{hits}}{\text{hits} + \text{misses}}$ per stage
\item \textbf{Time Saved:} $\sum(\text{expected\_compute\_time} - \text{cache\_load\_time})$
\item \textbf{Speedup Factor:} $\frac{\text{cold\_start\_time}}{\text{cached\_run\_time}}$
\item \textbf{Storage Overhead:} Total cache size in MB/GB
\item \textbf{Cache Efficiency:} The metric $\eta$ measures seconds of compute
      saved per MB of cache stored:
      \begin{equation}
        \eta = \frac{T_{\text{cold}} - T_{\text{warm}}}{\text{cache size per fold (MB)}}
        \quad \text{[s/MB]}
      \end{equation}
      Higher $\eta$ means more compute recovered per MB stored.
      The thresholds used in this thesis were calibrated empirically: after
      measuring all 15 model types, the viable cluster (boosting methods, SVMs,
      linear models) fell in the range $\eta \approx 4$--$2{,}500$~s/MB,
      while the non-viable cluster (tree ensembles, KNN) fell in the range
      $\eta \approx 0.19$--$0.4$~s/MB. The gap between clusters spans nearly a
      decade on a log scale, making the exact position of the boundary
      non-critical. The thresholds were set to bracket this gap:
      \begin{itemize}
        \item $\eta > 2$~s/MB $\Rightarrow$ \textbf{VIABLE}: each MB stored saves
              over 2 seconds of training; loading overhead is negligible.
        \item $0.5 \leq \eta \leq 2$~s/MB $\Rightarrow$ \textbf{BORDERLINE}:
              depends on dataset size and storage budget; evaluate per use case.
        \item $\eta < 0.5$~s/MB $\Rightarrow$ \textbf{NOT\_VIABLE}: each MB stored
              saves less than half a second; I/O overhead consumes a significant
              fraction of the benefit on smaller datasets or faster hardware.
      \end{itemize}
      To put these values in context: on a typical consumer SSD
      (read throughput $\approx 400$~MB/s), loading 1~MB takes
      $\approx 2.5$~ms. At $\eta = 2$~s/MB, that 1~MB loaded in 2.5~ms
      recovers 2 full seconds of training --- an $800{\times}$ net gain
      after I/O cost. Even at the NOT\_VIABLE boundary $\eta = 0.5$~s/MB,
      caching still yields a $200{\times}$ ratio of compute recovered to
      I/O cost; the verdict reflects the \textit{practical} concern that
      speedup margins erode on smaller datasets, not that caching is
      net-negative in absolute terms.
      These boundaries are applied consistently in Table~\ref{tab:cache_viability}
      and Figures~\ref{fig:cache_efficiency} and~\ref{fig:dual_metrics}.

\item \textbf{Relative Cache Efficiency:} A complementary metric
      $\rho = (T_{\text{cold}}/T_{\text{warm}}) / \text{size (MB)}$ [$\times$/MB]
      captures the proportional speedup per MB stored; $\rho > 1$~$\times$/MB
      agrees with the $\eta > 2$~s/MB viability threshold across all 15
      evaluated model types.

\item \textbf{Cache Viability Verdict:} A model is classified as
      \textbf{VIABLE} when $\eta > 2$~s/MB and $\rho > 1$~$\times$/MB.
      It is \textbf{NOT\_VIABLE} when $\eta < 0.5$~s/MB, indicating
      that warm-load time approaches cold-train time.
      The two metrics agree on all 15 evaluated model types.
\end{itemize}

\subsection{Secondary Metrics: Classification Performance}

Classification metrics validated that the caching framework produced correct results:

\begin{itemize}
\item \textbf{Accuracy:} Overall correct classifications
\item \textbf{Macro F1-Score:} Mean F1 across all classes (handles class imbalance)
\item \textbf{Cohen's Kappa:} Agreement beyond chance
\item \textbf{Per-Class F1:} Identifies problematic classes (especially N1)
\item \textbf{Confusion Matrix:} 5$\times$5 aggregated across all LOSO folds
\end{itemize}

\subsection{Reproducibility Metrics}

Reproducibility is validated at three levels: fingerprint determinism
(identical configurations produce identical SHA-256 hashes across multiple
independent runs), result consistency (identical fingerprints produce
bitwise-identical predictions and metrics within floating-point tolerance),
and resumability (an interrupted pipeline restarted from cache yields the
same final results as an uninterrupted run). All three properties were
confirmed across all 18 configurations in the full-scale experiment.

\textbf{Cold/warm correctness:} Empty cache (24 misses) followed by identical

\subsection{Verification Results}

On the 3-subject verification grid (24 cache operations), the cold start
produced 24 misses (0\% hit rate) and the subsequent warm re-run retrieved
all 24 cached models (100\% hit rate). Speedup measurements are reported
for the full-scale evaluation in Section~\ref{sec:results_caching}.

\subsection{Correctness Validation}

Beyond hit/miss behaviour, the verification suite programmatically validated
cache correctness across all 24 operations (3 subjects $\times$ 8 configurations,
cold and warm runs compared automatically):

\begin{itemize}
\item \textbf{Prediction Identity:} Cached models produced bitwise-identical
      predictions to freshly trained models, verified programmatically via
      \verb|numpy.array_equal(cold_preds, warm_preds)| for each of the 24
      configuration--subject pairs.
\item \textbf{Metric Reproducibility:} Accuracy, F1, and Cohen's $\kappa$ scores
      matched to floating-point precision ($<10^{-10}$ difference), computed
      independently for both runs and compared automatically.
\item \textbf{Zero Corruption:} All 24 pickle save/load cycles completed without
      errors; file integrity was confirmed by successful deserialization and
      inference on the held-out test set.
\item \textbf{Fingerprint Uniqueness:} All 12 configurations generated distinct
      SHA-256 fingerprints (zero hash collisions), verified by asserting
      uniqueness of the fingerprint set.
\end{itemize}

\subsubsection{Cache Scaling with Dataset Size}
\label{sec:results_scaling}

SVM models are used here because their well-characterised training complexity---O($n$) for linear kernels, O($n^2$) for RBF---provides a theoretically grounded test case: as subjects increase, training cost grows predictably while cache loading stays constant, isolating the scaling effect cleanly.

Cache speedup increases with dataset size for models whose training cost scales
super-linearly. SVM-RBF illustrates this cleanly: its O($n^2$) training complexity
causes cold run time to grow quadratically with subjects, while warm loading
remains constant, producing a monotonically growing speedup from 9$\times$
(10 subjects) to 201$\times$ (128 subjects). SVM-Linear, being O($n$), shows
approximately stable speedup across scales (1{,}736$\times$ to 3{,}093$\times$
with minor measurement variance at the 30-subject point), confirming that the
cache loading cost remains negligible relative to even linear training costs
at dataset scales above 30 subjects.

The same principle holds for the primary models when the feature space grows.
XGBoost training time scales with the number of input features: cold run time
increases from 855 seconds (30 features) to 3{,}683 seconds (149 features), a
4.3$\times$ increase in training cost. Cache loading remains constant at
approximately 7 seconds regardless of feature count, so the speedup grows
from 123$\times$ to 498$\times$ across the same range. Random Forest exhibits
the same directional pattern (12$\times$ to 110$\times$) but with warm-load times
of 288--356 seconds that do not scale with features, reflecting serialization cost
rather than computational cost. This confirms that caching becomes progressively
more valuable as model complexity or feature dimensionality increases---the
framework is not a fixed-speedup tool but one whose benefit grows with the
scale of the experiment.

\section{Findings}
\label{sec:findings}
% ============================================================================

Four key results are highlighted here; full interpretation is in
Chapter~\ref{chap:summary}.

\begin{itemize}
\item \textbf{Speedup:} XGBoost 37.5--202$\times$ (median 54$\times$);
      RF 11--23.5$\times$ (median 11.6$\times$); Layer~1 feature cache
      227$\times$. All 2{,}304 warm operations achieved 100\% cache hit rate.

\item \textbf{Reproducibility:} \texttt{accuracy\_match = True} for all
      18 configurations and 128 LOSO folds; zero fingerprint collisions.

\item \textbf{Viability spectrum:} 11 of 15 model types are cache-viable
      ($\eta > 2$~s/MB). The 4 not-viable types (RF, Extra Trees, KNN)
      share GB-scale serialized sizes. Verdicts are hardware-portable;
      speedup magnitudes are not.

\item \textbf{Scaling and selection:} Speedup grows with dataset size
      (SVM-RBF: 9$\times$ at 10 subjects to 201$\times$ at 128 subjects).
      Global ANOVA feature selection introduces $\leq 1.1\%$ accuracy difference
      vs.\ per-fold while retaining full warm-run cache speedup.
\end{itemize}

\section{Discussion}
\label{sec:discussion}
% ============================================================================

All four discussion topics below follow from one fundamental constraint:
\textbf{cache viability is governed by the ratio of serialized model size to
cold-train time.} A model whose load time grows proportionally to its train
time will never become more viable at larger scale --- size and savings cancel
out. Only models where train time grows \textit{faster} than load time (due to
super-linear training complexity or constant-size serialization) improve with
scale. Conversely, if the speedup ratio does not increase faster than cache
size grows with more data or features, caching cannot become structurally
viable regardless of hardware. This principle predicts the VIABLE/NOT\_VIABLE
verdicts before a single measurement is taken; the empirical results confirm it.

\subsection{XGBoost vs.\ Random Forest: A Multi-Dimensional Tradeoff}

XGBoost outperforms Random Forest on every measured dimension in this study:
higher classification accuracy (85.5\% vs.\ 81.9\%), higher cache speedup
(median 54$\times$ vs.\ 11.6$\times$), and dramatically smaller cache storage
(185~MB vs.\ 18.5~GB). This result challenges the common assumption that
Random Forest and XGBoost are interchangeable ensemble alternatives. For
LOSO-style experimentation with repeated runs, XGBoost is the clearly
preferable choice, not merely for accuracy, but for practical workflow efficiency.
Additionally, combining Mutual Information feature selection with Random Forest
would compound both bottlenecks: MI is 174$\times$ slower than ANOVA and RF is
borderline cache-viable, making such a pipeline doubly impractical for iterative
experimentation.

\subsection{Why Tree Ensembles and KNN Are Not Cache-Viable}

A model is classified as \textbf{NOT\_VIABLE} when its warm-load time approaches
or exceeds its cold-train time on the target hardware, negating the caching benefit,
or when the serialized model size makes repeated storage impractical.
Formally: caching is viable when $T_{\text{load}} \ll T_{\text{train}}$
and the per-configuration disk footprint is manageable.

The NOT\_VIABLE verdict for Random Forest, Extra Trees, and KNN stems from
a fundamental serialization characteristic. Tree-ensemble models store an
explicit structure for each decision tree: hundreds or thousands of node
conditions, feature indices, and threshold values. A 100-tree Random Forest
trained on 149 features with 127 training subjects produces a model file of
600--1544~MB. Loading this from disk takes 13--23 seconds per fold, which
approaches or exceeds the training time. KNN stores the entire training dataset
in memory as the ``model'', resulting in similarly large serialized sizes.

In contrast, gradient boosting models (XGBoost, GBM, LightGBM) use
regularisation and shallow trees (often depth 3--6) that serialise to compact
representations of 1--8~MB, loading in under one second.

\subsection{Storage vs.\ Speed Tradeoff}

The fundamental tradeoff is: $T_{\text{load}}(M) = \text{size}(M) / B_{\text{disk}}$,
where $B_{\text{disk}}$ is disk read throughput. Models with small serialized sizes
load in milliseconds and are unambiguously viable; models with large sizes
(100s of MB to GB per fold) have load times that approach training time, and viability
depends on hardware and how many re-runs are planned.

The framework exposes a clear storage-speed tradeoff that depends on the
experimental context. For XGBoost ($\sim$185~MB per 9-config run), caching
is unambiguously beneficial. For Random Forest ($\sim$18.5~GB per 9-config
run), the decision depends on available storage and the number of expected
re-runs. A single re-run saves approximately 9 hours of computation at the
cost of 18.5~GB of disk space. For a researcher who will run 3--4 parameter
sweeps, this is clearly worthwhile. For a one-time experiment, it may not be.

cost-effective.

\subsection{Feature Selection Scope: Accuracy vs.\ Cache Efficiency}
\label{sec:discussion_scope}

A practical design choice in any LOSO caching workflow is whether feature
selection runs \textit{globally} (once per configuration, fitted on all subjects)
or \textit{per-fold} (refitted independently inside each LOSO iteration).
Per-fold selection is statistically stricter --- it eliminates the minor
label leakage that arises when the held-out subject's labels contribute to
the global feature ranking --- but it has a significant cache efficiency cost.

Global feature selection (used in this thesis) introduces a structural leakage
of $1/128 \approx 0.8\%$ of selection data per fold. An empirical benchmark
(Table~\ref{tab:global_vs_fold} and Figure~\ref{fig:global_vs_perfold}) measured
this effect directly: the mean accuracy difference across all configurations and
subject counts is $\pm 0.4\%$ with a maximum of $1.1\%$, and the direction is
inconsistent --- per-fold occasionally outperforms global, confirming the effect
is noise rather than systematic bias. The warm-run speedup benefit of global
scope is more significant: XGBoost retains its full 14--23$\times$ warm speedup,
whereas per-fold scope degrades to 4--9$\times$ because feature selection
re-runs for every fold even on cache hits. Global selection is therefore the
pragmatic choice for caching-oriented workflows.

\subsection{Limitations}
\label{sec:limitations}

\begin{itemize}
\item \textbf{Single dataset:} All experiments were conducted on the BOAS
      dataset (128 subjects). Generalisation of cache viability verdicts to
      other EEG datasets or non-EEG domains requires further validation.

\item \textbf{Global feature selection:} Feature selection is performed across
      all subjects before LOSO splits, introducing minor label leakage
      ($1/128 \approx 0.8\%$ of selection data per fold). An empirical benchmark
      showed the resulting accuracy difference is $\leq 1.1\%$ with no
      consistent direction (Section~\ref{sec:experiment_setup}). A strictly
      correct per-fold implementation would eliminate this leakage at the cost
      of reduced warm-run cache speedup.

\item \textbf{No deep learning comparison:} The framework was evaluated only
      on traditional ML models. Deep learning models (e.g., U-Sleep, CNNs)
      have different serialization characteristics and may exhibit different
      cache viability patterns.

\item \textbf{No multi-user evaluation:} The framework was designed for
      single-researcher workflows. Cache sharing across multiple concurrent
      users would require locking mechanisms not implemented here.

\item \textbf{Layer~1 fingerprint validation is lenient:} The feature cache
      (Layer~1) stores a preprocessing config fingerprint with each cached
      file, but loads it in lenient mode by default --- the fingerprint is
      recorded but not validated against the current config on retrieval.
      The fingerprint guarantee enforced by the \texttt{LOSOFingerprint} class
      applies to Layer~2 (the LOSO model cache), where strict SHA-256
      comparison is the primary contribution of this work. For Layer~1, the
      design rationale is that preprocessing parameters are fixed for the
      duration of a thesis project; activating strict Layer~1 validation
      requires setting \texttt{strict\_validation=True} in \texttt{feature\_cache.py}.
      The thesis claim ``changing preprocessing parameters invalidates the
      cache automatically'' therefore applies to Layer~2 only; Layer~1 caches
      created with legacy code (\texttt{config\_fingerprint="unknown"}) would
      not be invalidated until manually deleted or regenerated.

\item \textbf{Configuration space governance:} Because every unique parameter
      combination produces a distinct SHA-256 fingerprint, an unconstrained
      parameter space yields thousands of distinct cache keys each populated
      at most once, delivering zero reuse. The framework is technically correct
      in this scenario but provides no speedup. Effective use requires an
      agreed set of parameter values before experimentation begins; the
      fingerprinting mechanism then guarantees that any re-run within that
      grid is fully cached. This governance responsibility sits with the
      research team, not the framework.

\item \textbf{Thermal throttling on resource-constrained hardware:}
      Cold-run timing measurements on sustained workloads (e.g., the 9-hour
      RF 149-feature run) may be inflated by CPU thermal throttling.
      The measured speedup ratios are therefore conservative upper bounds on
      throttled hardware; on a temperature-controlled workstation the cold-run
      time would be lower and the speedup ratio smaller, though the
      VIABLE/NOT\_VIABLE verdict is unchanged because it depends on model
      serialization size, not absolute timings.
\end{itemize}
