\section{Theoretical Framework}
\label{sec:theoretical}

The framework is grounded in three key concepts: configuration fingerprinting, a distinction between persistent storage and intelligent caching, and two orthogonal evaluation dimensions.

\subsection{Configuration Fingerprinting}

Configuration fingerprinting provides the foundation for cache key generation and reproducibility tracking. Each experiment configuration is uniquely identified by a cryptographic hash computed as:

\begin{equation}
\text{fingerprint} = \text{SHA256}(\text{canonical\_json}(\text{config}))[:32]
\end{equation}

The fingerprint uses 32 hexadecimal characters (128 bits), providing strong collision resistance suitable for large-scale experiment tracking. The configuration dictionary includes:

\begin{itemize}
    \item \textbf{Random seed:} Ensures deterministic behavior (e.g., \texttt{42})
    \item \textbf{Code version:} Git commit or semantic version (e.g., \texttt{v1.0}, \texttt{git:abc123})
    \item \textbf{Model configuration:} Algorithm name and hyperparameters (e.g., \texttt{xgboost}, \texttt{max\_depth: 6})
    \item \textbf{Feature configuration:} Base feature count, correlation threshold, and top-K selection
    \item \textbf{Held-out subject ID:} Critical for LOSO---prevents cross-fold contamination (e.g., \texttt{SC4001})
\end{itemize}

An example fingerprint: \texttt{a3f2b8c1d4e5f6a7b8c9d0e1f2a3b4c5}.

The fingerprint generation module is fully implemented, providing the \texttt{LOSOFingerprint} class with methods for generating fingerprints from configuration objects or dictionaries. The \texttt{generate\_cache\_key()} function provides a convenient interface that combines the fingerprint with the held-out subject ID (e.g., \texttt{a3f2b8c1...\_SC4001}) for direct use as cache file names.

\subsection{Configuration-Independent vs. Configuration-Dependent Caching}
\label{sec:storage_vs_caching}

A fundamental distinction in this framework is between \textit{configuration-independent caching} (artifacts that depend only on the dataset) and \textit{configuration-dependent caching} (artifacts that require fingerprint-based invalidation when experimental parameters change). While both avoid redundant computation, they serve different purposes:

\textbf{Configuration-Independent Cache} refers to artifacts that are computed once per dataset version and remain valid across all experimental variations. These artifacts do not depend on model hyperparameters, feature selection criteria, or cross-validation strategies. In this framework, the extracted feature matrices constitute configuration-independent caching---they are computed once per subject and reused across all experiments.

\textbf{Configuration-Dependent Cache} refers to artifacts whose validity depends on experimental configuration. These artifacts require fingerprint-based invalidation when parameters change. The LOSO model cache, which stores trained models keyed by configuration fingerprint and held-out subject, exemplifies configuration-dependent caching.

\subsubsection{Layer 1: Feature Extraction as Persistent Storage (Implemented)}

The feature extraction layer computes \textbf{all 149 features} for each subject across 6 EEG channels (F3, F4, C3, C4, O1, O2) and stores them in compressed NPZ format. This is classified as persistent storage because:

\begin{itemize}
    \item Features depend only on the raw EEG data and fixed preprocessing parameters
    \item Changing model hyperparameters or feature selection thresholds does not invalidate the stored features
    \item The full feature set is extracted once; downstream selection operates on this cached superset
\end{itemize}

In this implementation, the pipeline checks for \texttt{\{subject\_id\}.npz}
before extracting; if found, it is loaded directly and extraction is skipped.
This layer is fully implemented and provides measured speedups of approximately 227$\times$ for feature loading compared to re-extraction.

\subsubsection{Layer 2: LOSO Model Caching (Primary Contribution)}
\label{sec:loso_caching}

The LOSO model cache represents the \textbf{primary contribution} of this thesis.
The complete system---including fingerprint generation, cache storage, retrieval
logic, and automatic invalidation---is fully implemented and validated through
comprehensive testing (see Section~\ref{sec:verification}) and the full-scale
18-configuration experiment (see Chapter: Results and Discussion).

\textbf{Full-Scale Results:} The caching system achieved 100\% cache hit rate
across all 18 configurations and 128 LOSO folds (2{,}304 total operations).
Measured speedups for XGBoost ranged from 122$\times$ to 498$\times$ (median
178$\times$) with cache sizes of $\sim$185~MB. Random Forest achieved 12--110$\times$
speedup (median 17.7$\times$) but required $\sim$18.5~GB of cache storage.

\textbf{Why LOSO-Level Caching, Not Just Model Caching?}

A critical design decision is caching at the LOSO fold level rather than caching a single global model. This distinction is essential:

\begin{enumerate}
    \item \textbf{Data Leakage Prevention:} Each LOSO fold has a unique fingerprint that includes the \texttt{held\_out\_subject} field. A model trained with Subject~A held out cannot be retrieved when Subject~B is the test subject---the fingerprints differ (as demonstrated by the implemented \texttt{LOSOFingerprint} class), forcing a cache miss and retraining.
    
    \item \textbf{Scientific Validity:} In LOSO cross-validation, each fold produces a different model because the training data differs. Caching a single ``global model'' would be scientifically invalid---there is no single model, but rather $N$ distinct models (one per subject held out).
    
    \item \textbf{Incremental Experimentation:} When adding new subjects or modifying experimental parameters, only affected folds require retraining. Cached models for unchanged configurations remain valid and are reused automatically.
\end{enumerate}

\textbf{Why Model-Level Caching, Not Result-Level Caching?}

Within each LOSO fold, the framework caches the trained model rather than just the evaluation metrics:

\begin{itemize}
    \item \textbf{Post-hoc Analysis:} Cached models enable feature importance analysis, prediction on new data, or computation of additional metrics without retraining.
    \item \textbf{Reproducibility Verification:} Identical fingerprints guarantee identical model states, enabling verification that predictions (not just aggregate metrics) are reproducible.
    \item \textbf{Debugging Capability:} When unexpected results occur, the exact model state can be inspected rather than requiring reproduction of the entire training process.
\end{itemize}

The trade-off is increased storage (models are larger than metric summaries), but this is acceptable given the scientific benefits and modern storage capacities.

\subsection{Two Orthogonal Evaluation Dimensions}

The framework evaluation follows two orthogonal dimensions that structure the experimental analysis:

\begin{itemize}
    \item \textbf{WHY dimension (Optimization Goals):}
    \begin{itemize}
        \item Latency: Time to retrieve cached vs. computed results
        \item Throughput: Subjects processed per unit time
        \item Time Saved: Cumulative computation time avoided across experimental iterations
    \end{itemize}
    \item \textbf{WHERE dimension (Cached Artifact Levels):}
    \begin{itemize}
        \item Layer 1: Extracted feature matrices (persistent storage---implemented)
        \item Layer 2: Trained LOSO fold models (intelligent cache--implemented)
        \item Results: Final metrics (always recomputed from cached models for verification)
    \end{itemize}
\end{itemize}

This two-dimensional framework enables systematic evaluation of caching benefits across different experimental scenarios, from simple re-runs (maximum cache hits) to parameter sweeps (partial cache hits) to completely new configurations (cache misses establishing new baselines).

\section{Data Validation and Integrity Verification}
\label{sec:data_validation}

Prior to feature extraction, each recording undergoes an automated 12-point
validation protocol covering annotation integrity (sufficient epochs, all
5 stages present, duration $\geq$2~hours, filter rate $<$30\%, temporal
continuity), structural consistency (correct channel count and sampling rate),
EDF-annotation alignment (onset bounds, sample bounds), and signal quality
(no NaN/$\pm\infty$ values, no flat channels, amplitude $<10^6~\mu$V).

All 128 BOAS subjects passed all checks. The dataset exhibits minimal
artifact contamination ($<$0.1\% epoch exclusion rate). The dataset
creators report a human-AI agreement of 87.08\% for PSG
data~\cite{esparza2026automatic}, which serves as a performance benchmark
in this thesis.




\section{Preprocessing Pipeline}
\label{sec:preprocessing}

The preprocessing pipeline transforms raw EDF recordings into clean, epoched signals suitable for feature extraction.

\subsection{Signal Filtering}

The following signal conditioning is applied sequentially to each EEG channel:

\begin{enumerate}
\item \textbf{Bandpass Filter:} A 5th-order Butterworth filter with cutoff frequencies of 0.5--40 Hz removes DC drift and high-frequency noise while preserving sleep-relevant frequency bands.
\item \textbf{Notch Filter:} A notch filter at 50 Hz (Q=30) removes power line interference.
\item \textbf{Downsampling:} 256 Hz $\rightarrow$ 128 Hz using decimation with built-in anti-aliasing low-pass filter (prevents aliasing artifacts).
\end{enumerate}

All preprocessing parameters are included in the cache fingerprint, ensuring automatic invalidation when parameters change.

\subsection{Epoch Creation}

Continuous signals are segmented into 30-second epochs aligned with hypnogram annotations. Each epoch contains 3,840 samples (30 seconds $\times$ 128 Hz) across 6 channels, resulting in an array of shape $(n_\text{epochs}, 3840, 6)$ per subject.

\subsection{Quality Control}

Epochs are filtered according to the comprehensive validation criteria established in Section~\ref{sec:data_validation}. In summary, epochs must satisfy:
\begin{itemize}
\item Amplitude within $\pm$500 $\mu$V (reject artifacts)
\item No NaN or infinite values
\item Valid sleep stage annotation (exclude disconnection events, artifacts, and unknown stages)
\end{itemize}

The full 12-point validation protocol ensures data integrity before feature extraction begins.

\subsection{Feature Engineering Strategy and Caching Prioritization}

While Satapathy et al.\ (2024) demonstrate high performance using a set of 63 multi-modal features~\cite{satapathy2024multimodal}, this thesis implements an expanded set of 149 EEG-specific features. This deliberate over-provisioning serves a dual purpose in evaluating the caching framework: first, it increases the computational cost of the feature extraction phase, providing a more rigorous stress test for the Tier~1 Persistent Feature Store; second, it enables the systematic evaluation of feature-count cut-offs as a configuration parameter. Because all 149 features are cached as a Materialised Intermediate Representation, the framework can test various sub-selections (e.g., top 20, 50, or 100 features) with near-zero latency, demonstrating the advantage of deterministic caching over traditional linear pipelines.

The feature extraction pipeline implements a comprehensive 149-feature representation spanning time-domain statistics, frequency-domain band powers, complexity measures, and cross-channel connectivity metrics. This feature set was deliberately designed to balance discriminative power for sleep stage classification with computational tractability for caching optimization.

While additional features such as sample entropy, permutation entropy, band power ratios, or spindle density detection could marginally enhance classification performance, the current architecture intentionally prioritizes caching efficiency as the central research contribution. By establishing a fixed, comprehensive feature space, the system enables complete cache validity across experimental variations in feature selection, model hyperparameters, and cross-validation strategies---changes that would otherwise trigger costly recomputation.

This design philosophy reflects a strategic trade-off: rather than pursuing incremental gains in classification accuracy through feature proliferation, the framework optimizes for reproducibility, computational efficiency, and scalability. The result is a system where subsequent experiments achieve near-instantaneous feature retrieval, reducing pipeline latency from hours to seconds, minimizing computational resource consumption, and ensuring deterministic reproducibility across research iterations. Future extensions may incorporate additional feature domains; however, the current implementation demonstrates that intelligent caching of a well-designed feature set delivers substantial practical benefits that outweigh marginal accuracy improvements from feature expansion.

\subsection{Feature Selection Strategy}

After extracting all 149 features, optional feature selection is applied using a two-stage pipeline:

\textbf{Stage 1: Correlation-Based Filtering}
\begin{itemize}
\item Removes highly correlated features using Pearson correlation
\item Threshold options: 0.75, 0.90, or None (no filtering)
\item Prevents multicollinearity in downstream models
\end{itemize}

\textbf{Stage 2: ANOVA F-statistic Selection}
\begin{itemize}
\item Ranks features by class-discriminative power using F-statistic (\texttt{f\_classif})
\item Top-K options: 30, 50, or None (retain all 149 features)
\item ANOVA selected over Mutual Information based on a pilot benchmark (10 subjects, 5-fold CV): ANOVA was 174$\times$ faster \textit{and} 3.68 percentage points more accurate, giving strict dominance with no trade-off (Table~\ref{tab:anova_vs_mi}).
\end{itemize}

\textbf{Implementation Note:} Feature selection is performed \textbf{globally}
(not per-fold) for three compounding reasons.
First, ANOVA (\texttt{f\_classif}) is a deterministic computation: given a fixed
dataset and fixed parameters, it always produces the same ranked feature list.
A single global fit therefore yields exactly one stable feature set that can be
reused as a consistent fingerprint component across all 128 LOSO folds, without
any per-fold bookkeeping.
Second, a single fit eliminates 127 redundant ANOVA runs per configuration,
reducing feature-selection overhead from $O(N)$ to $O(1)$ relative to fold
count, and ensuring all 18 model configurations operate on identical feature
subsets so that accuracy differences are attributable solely to the model.
Third, the empirical impact of the residual leakage is negligible: a controlled
benchmark (Figure~\ref{fig:global_vs_perfold}) measured an accuracy difference
of $\pm 0.4\%$ on average with a maximum of $1.1\%$ and no consistent direction,
confirming that the held-out subject's $1/128 \approx 0.8\%$ contribution to the
global ranking does not meaningfully inflate reported performance.

The global approach also aligns with realistic deployment conditions.
In production, a sleep-staging system is trained on all available historical
recordings and applied to new, unseen patients; the feature-selection step
naturally sees the full training corpus at fit time, making global selection
the operationally correct mode rather than a methodological shortcut.
The remaining leakage is a known and quantified limitation
(Section~\ref{sec:limitations}); Section~\ref{sec:future_work} describes a
\textit{lazy selection cache} that would eliminate it without sacrificing
the warm-run speedup.

\subsection{Feature Extraction Optimization}

To enable efficient processing of the BOAS dataset (128 subjects, $\sim$1000 epochs each, $\sim$128,000 total epochs), several optimization strategies were evaluated and implemented.

\subsubsection{Implemented Optimizations (No Quality Loss)}

\textbf{Multiprocess Parallelization:} Feature extraction was parallelized across CPU cores using Python's joblib library with process-based workers. Thread-based parallelization was initially attempted but provided no speedup due to Python's Global Interpreter Lock (GIL) blocking concurrent execution of CPU-bound NumPy/SciPy operations. Process-based parallelization achieved near-linear scaling, reducing per-epoch extraction time from $\sim$4.7s to $\sim$1.1s on an 8-core system (4.3$\times$ speedup).

\textbf{Optimized Complexity Measures:} The computationally expensive complexity features (Detrended Fluctuation Analysis, Hurst exponent, Hjorth parameters) were accelerated using the antropy library, which provides Numba JIT-compiled implementations of identical peer-reviewed algorithms. This optimization maintained mathematical equivalence while reducing complexity feature computation time by approximately 2--3$\times$.

\textbf{Feature Caching Strategy:} A comprehensive caching system was implemented to compute features once for all 6 EEG channels and store them in compressed NumPy archives. Subsequent experiments can load cached features and filter to the required channel subset at runtime, eliminating redundant computation when comparing channel configurations.

\subsubsection{Evaluated But Not Implemented (Would Reduce Quality)}

\textbf{Approximate Hurst Exponent:} Replacing the DFA-based Hurst exponent with faster auto\-correlation-based approximations was considered but rejected. While offering 5--10$\times$ speedup, this approach produces systematically different values (10--15\% deviation) and would compromise comparability with existing sleep staging literature that relies on standard DFA.

\textbf{Reduced DFA Scale Resolution:} Decreasing the number of window scales in DFA from 10 to 5 was evaluated. Although this would halve DFA computation time, it introduces 2--3\% variance in the resulting exponents, potentially affecting the discrimination of sleep stages with similar spectral profiles (particularly N2/N3 differentiation).

\textbf{Sample Entropy Approximation:} Fast approximate entropy measures were considered but not implemented, as sample entropy's $O(N^2)$ complexity was already addressed through parallelization, and approximate methods would alter the feature's discriminative properties for sleep stage classification.

\subsection{Fingerprint Generation}
\label{sec:fingerprint_generation}

Cache key generation follows a five-step process:

\begin{enumerate}
\item \textbf{Configuration assembly.} All tracked parameters
      (\texttt{random\_seed}, \texttt{code\_version}, \texttt{model\_config}
      with algorithm name and hyperparameters, \texttt{feature\_config} with
      base feature count, correlation threshold, top-K selection, and the
      sorted list of selected feature names, and \texttt{held\_out\_subject})
      are combined into a single Python dictionary.

\item \textbf{Canonical serialization.} The dictionary is serialized to
      JSON using \texttt{json.dumps()} with \texttt{sort\_keys=True} and
      compact separators \texttt{(',',':')}, ensuring a deterministic string
      representation regardless of insertion order.

\item \textbf{SHA-256 hashing.} The canonical JSON string is encoded to
      UTF-8 and hashed using SHA-256, producing a 256-bit (64 hexadecimal
      character) digest.

\item \textbf{Truncation.} The hash is truncated to 32 hexadecimal
      characters (128 bits) for filesystem compatibility, while retaining
      sufficient collision resistance for the experimental scale
      ($\sim$2{,}300 configurations).

\item \textbf{Cache key formation.} The truncated hash is concatenated
      with the held-out subject identifier to form the final cache key:
      \texttt{\{fingerprint\}\_\{subject\}}, e.g.,
      \texttt{a3f2b8c1d4e5f6a7...\_sub-001}.
\end{enumerate}

The corresponding implementation in \texttt{fingerprint.py}:

\begin{verbatim}
canonical = json.dumps(config.to_dict(),
                       sort_keys=True, separators=(',',':'))
fingerprint = hashlib.sha256(canonical.encode()).hexdigest()[:32]
cache_key = f"{fingerprint}_{held_out_subject}"
\end{verbatim}

\textbf{Partial cache reuse across experiment sweeps.} Configurations sharing
identical hyperparameters and subject folds produce matching fingerprints,
enabling automatic reuse. For example, sweeps $\{A_1, A_2, B_1, B_2\}$ and
$\{B_1, B_2, C_1, C_2\}$ reuse cached $B_1$, $B_2$ models while training
$C_1$, $C_2$ fresh. This was verified in
\texttt{experiment\_b\_fingerprint\_invalidation} of
\texttt{run\_thesis\_benchmark.py}.

\textbf{Version control safety.} The \texttt{code\_version} parameter
(set to the current semantic version or Git commit hash) ensures that
implementation changes automatically invalidate the cache. If an implementation
migration introduces a bug or modifies model behaviour, the resulting fingerprint
differs from previous versions, preventing retrieval of stale cached
models.

\subsection{Cache Storage Structure}

Table~\ref{tab:cache_structure} summarises the two-layer directory layout.

\subsection{Cache Hit/Miss Logic}

For each LOSO fold, the system generates the configuration fingerprint and
checks whether \verb|{fingerprint}_{subject}.joblib| exists on disk. On a
\textbf{cache hit}, the model is loaded (XGBoost: $\sim$58~ms; RF:
$\sim$2.5~s). On a \textbf{cache miss}, the model is trained (XGBoost:
$\sim$29~s; RF: $\sim$248~s) and saved to disk for future reuse.

This single-tier disk-based approach is well-suited to the batch-oriented
experimental workflow, where each model is accessed once per run.


\textbf{Note on Two-Tier RAM+Disk Caching:} An in-memory caching layer
(LRU cache in RAM, backed by disk) was considered but not implemented. The
batch-oriented experimental workflow accesses each model exactly once per run,
so the RAM layer would not achieve repeated hits. Additionally, preliminary
testing indicated that the performance bottleneck is disk I/O throughput rather
than seek latency---an in-memory tier did not meaningfully reduce loading times
for the model sizes encountered. A detailed discussion of two-tier caching as
a future extension is provided in Chapter~5 (Future Work).

\subsection{Handling Data Changes and Cache Invalidation}

A critical challenge in ML experiment caching is ensuring cache validity when underlying data changes. Our implementation addresses this through a multi-layered integrity system embedded in the cache metadata. Each cached entry stores a configuration fingerprint---a SHA-256 hash of all preprocessing parameters, alongside explicit version identifiers (\texttt{data\_version} and \texttt{feature\_version}) and source file metadata.

When raw data modifications occur, three invalidation strategies are available:

\begin{enumerate}
\item \textbf{Version Bumping:} Incrementing the \texttt{data\_version} parameter in the configuration file causes the fingerprint to change, automatically invalidating all cached entries without requiring manual intervention. This approach preserves the cache structure while ensuring fresh computation.

\item \textbf{Cache Purging:} For significant data changes or version migrations, the global cache directory can be deleted entirely, triggering a complete recomputation on the next execution.

\item \textbf{Strict Validation:} An optional \texttt{strict\_validation} mode computes file checksums during cache lookup, detecting any byte-level changes to source files. While more computationally expensive, this provides cryptographic guarantees of data integrity.
\end{enumerate}

Our implementation defaults to version-based validation (Option 1), balancing integrity assurance with computational efficiency. The fingerprint system ensures that configuration drift---such as modified filter parameters or channel selections---automatically triggers recomputation, while unchanged configurations benefit from cached results. This approach aligns with reproducibility requirements in scientific computing, where identical inputs must yield identical outputs.

\subsection{Implementation Challenge: Incomplete Fingerprinting}
\label{sec:implementation_challenges}

During development, a feature mismatch bug was identified: models trained
with 37 features crashed during loading when feature selection produced 33
features under identical selection parameters but with a different
random seed. The root cause was that the fingerprint included the selection
parameters (correlation threshold, top-K count) but not the resulting feature
names. The fix was to hash the sorted list of selected feature names into the
fingerprint, ensuring that any change in the actual feature set invalidates
the cache. This bug was detected through early verification testing
(Section~\ref{sec:verification}) before full-scale experiments.

\subsection{Cross-Validation Strategy}

\textbf{Leave-One-Subject-Out (LOSO)} cross-validation is the primary evaluation strategy:
\begin{itemize}
\item 128 folds (one per subject)
\item Each fold: 127 subjects for training, 1 subject for testing
\item Ensures true cross-subject generalization (no data leakage)
\item Total model trainings: 18 configurations $\times$ 128 folds = 2,304 trainings
\end{itemize}

\subsection{Caching Evaluation Scenarios}

Five scenarios were executed to evaluate caching performance:

\begin{enumerate}
\item \textbf{First Run (Cold Start):} All cache misses, establishes baseline timing
\item \textbf{Identical Re-run:} All cache hits, measures maximum speedup
\item \textbf{Changed Preprocessing:} Invalidates Stage 1 and Stage 2 caches
\item \textbf{Changed Feature Selection:} Stage 1 and Stage 2 hit (selection at runtime)
\item \textbf{Add New Subjects:} Existing subjects hit, new subjects miss
\end{enumerate}

\subsection{Correctness Validation}

Beyond hit/miss behaviour, the verification suite programmatically validated
cache correctness across all 24 operations (3 subjects $\times$ 8 configurations,
cold and warm runs compared automatically):

\begin{itemize}
\item \textbf{Prediction Identity:} Cached models produced bitwise-identical
      predictions to freshly trained models, verified programmatically via
      \verb|numpy.array_equal(cold_preds, warm_preds)| for each of the 24
      configuration--subject pairs.
\item \textbf{Metric Reproducibility:} Accuracy, F1, and Cohen's $\kappa$ scores
      matched to floating-point precision ($<10^{-10}$ difference), computed
      independently for both runs and compared automatically.
\item \textbf{Zero Corruption:} All 24 pickle save/load cycles completed without
      errors; file integrity was confirmed by successful deserialization and
      inference on the held-out test set.
\item \textbf{Fingerprint Uniqueness:} All 12 configurations generated distinct
      SHA-256 fingerprints (zero hash collisions), verified by asserting
      uniqueness of the fingerprint set.
\end{itemize}

\subsection{Invalidation Testing}

Cache invalidation was verified by deliberately modifying individual
configuration parameters and asserting that the framework correctly triggered
a cache miss (i.e., retrained from scratch) in each case:

\begin{itemize}
\item \textbf{Changed Hyperparameter:} Modifying \verb|max_depth| from 6 to 8
      produced a different fingerprint and triggered a cache miss, confirming
      that hyperparameter changes invalidate the cache.
\item \textbf{Changed Feature Selection:} Changing top-K from 30 to 50 produced
      a different fingerprint and triggered a cache miss, confirming that
      feature configuration changes invalidate the cache.
\item \textbf{Changed Held-Out Subject:} Running sub-1 vs.\ sub-2 with identical
      model parameters produced different fingerprints and separate cache entries,
      confirming that no cross-fold contamination occurs.
\end{itemize}

In each test, the fingerprint difference was verified programmatically by
comparing the SHA-256 hashes before and after the parameter modification.
These results confirmed that the caching logic correctly distinguishes
configurations at the individual-parameter level and justified proceeding
to the full 2{,}304-operation experiment. The full-scale caching results
are presented in Section~\ref{sec:results_caching}.


\section{Evaluation Metrics}
\label{sec:evaluation_plan}

The experimental evaluation used three categories of metrics, corresponding to
the primary caching contribution, the secondary classification validation, and
the reproducibility requirements.

\subsection{Primary Metrics: Caching Efficiency}

The primary thesis contribution was evaluated through caching metrics:

\begin{itemize}
\item \textbf{Cache Hit Rate:} $\frac{\text{hits}}{\text{hits} + \text{misses}}$ per stage
\item \textbf{Time Saved:} $\sum(\text{expected\_compute\_time} - \text{cache\_load\_time})$
\item \textbf{Speedup Factor:} $\frac{\text{cold\_start\_time}}{\text{cached\_run\_time}}$
\item \textbf{Storage Overhead:} Total cache size in MB/GB
\end{itemize}

\subsection{Secondary Metrics: Classification Performance}

Classification metrics validated that the caching framework produced correct results:

\begin{itemize}
\item \textbf{Accuracy:} Overall correct classifications
\item \textbf{Macro F1-Score:} Mean F1 across all classes (handles class imbalance)
\item \textbf{Cohen's Kappa:} Agreement beyond chance
\item \textbf{Per-Class F1:} Identifies problematic classes (especially N1)
\item \textbf{Confusion Matrix:} 5$\times$5 aggregated across all LOSO folds
\end{itemize}

\subsection{Reproducibility Validation}

Reproducibility was verified through:

\begin{itemize}
\item \textbf{Fingerprint Determinism:} Same config $\rightarrow$ same fingerprint (verified across multiple runs)
\item \textbf{Result Consistency:} Same fingerprint $\rightarrow$ identical metrics (within floating-point tolerance)
\item \textbf{Resumability:} Interrupted pipeline $\rightarrow$ resumed from cache $\rightarrow$ identical final results
\end{itemize}

at dataset scales above 30 subjects. Figures~\ref{fig:cache_scaling} and~\ref{fig:svm_scaling} visualise
this scaling behaviour.

The same principle holds for the primary models when the feature space grows.
XGBoost training time scales with the number of input features: cold run time
increases from 855 seconds (30 features) to 3{,}683 seconds (149 features), a
4.3$\times$ increase in training cost. Cache loading remains constant at
approximately 7 seconds regardless of feature count, so the speedup grows
from 123$\times$ to 498$\times$ across the same range. Random Forest exhibits
the same directional pattern (12$\times$ to 110$\times$) but with warm-load times
of 288--356 seconds that do not scale with features, reflecting serialization cost
rather than computational cost. This confirms that caching becomes progressively
more valuable as model complexity or feature dimensionality increases---the
framework is not a fixed-speedup tool but one whose benefit grows with the
scale of the experiment.

\section{Findings}
\label{sec:findings}
% ============================================================================

Six claims formulated at the outset of this thesis were validated through
systematic experimentation:

\begin{enumerate}
\item \textbf{Cache provides significant LOSO speedup.}
      XGBoost: 122--498$\times$ (median 178$\times$).
      SVM-Linear: 1{,}550--3{,}093$\times$ scaling with dataset size.
      The framework delivers order-of-magnitude reductions in experiment time.

\item \textbf{Cached models produce identical results.}
      \texttt{accuracy\_match = True} for all 18 configurations. Fingerprint-based
      cache keys guarantee that retrieved models correspond exactly to the
      requested configuration, ensuring scientific validity.

\item \textbf{ANOVA is strictly superior to Mutual Information for this task.}
      174$\times$ faster with $+$3.68\% higher accuracy. Justified by the
      linear discriminative structure of EEG frequency-domain features.

\item \textbf{Cache benefit scales with dataset size.}
      SVM-RBF speedup grows from 9$\times$ (10 subjects) to 201$\times$
      (128 subjects), because training cost grows quadratically while cache
      loading remains constant. The framework becomes more valuable as studies scale.

\item \textbf{A clear model viability spectrum exists.}
      11 of 15 models are cache-viable. Boosting-based models (GBM, XGBoost,
      AdaBoost) achieve extreme speedups with compact cache sizes. Four model
      types (Random Forest, Extra Trees, KNN $k=5$, KNN $k=10$) are not viable
      due to GB-scale serialized sizes.

\item \textbf{Viability verdicts are hardware-portable.}
      The same models are VIABLE/NOT\_VIABLE regardless of machine. Speedup
      magnitudes differ between the development laptop and the desktop machine,
      but the ranking is consistent, confirming that the framework's behaviour
      generalises across hardware configurations.
\end{enumerate}

\section{Discussion}
\label{sec:discussion}
% ============================================================================

\subsection{XGBoost vs.\ Random Forest: A Multi-Dimensional Tradeoff}

XGBoost outperforms Random Forest on every measured dimension in this study:
higher classification accuracy (84.9\% vs.\ 81.5\%), higher cache speedup
(178$\times$ vs.\ 17.7$\times$), and dramatically smaller cache storage
(185~MB vs.\ 18.5~GB). This result challenges the common assumption that
Random Forest and XGBoost are interchangeable ensemble alternatives. For
LOSO-style experimentation with repeated runs, XGBoost is the clearly
preferable choice, not merely for accuracy, but for practical workflow efficiency.

\subsection{Why Tree Ensembles and KNN Are Not Cache-Viable}

The NOT\_VIABLE verdict for Random Forest, Extra Trees, and KNN stems from
a fundamental serialization characteristic. Tree-ensemble models store an
explicit structure for each decision tree: hundreds or thousands of node
conditions, feature indices, and threshold values. A 100-tree Random Forest
trained on 149 features with 127 training subjects produces a model file of
600--1544~MB. Loading this from disk takes 13--23 seconds per fold, which
approaches or exceeds the training time. KNN stores the entire training dataset
in memory as the ``model'', resulting in similarly large serialized sizes.

In contrast, gradient boosting models (XGBoost, GBM, LightGBM) use
regularisation and shallow trees (often depth 3--6) that serialise to compact
representations of 1--8~MB, loading in under one second.

\subsection{Storage vs.\ Speed Tradeoff}

The framework exposes a clear storage-speed tradeoff that depends on the
experimental context. For XGBoost ($\sim$185~MB per 9-config run), caching
is unambiguously beneficial. For Random Forest ($\sim$18.5~GB per 9-config
run), the decision depends on available storage and the number of expected
re-runs. A single re-run saves approximately 9 hours of computation at the
cost of 18.5~GB of disk space. For a researcher who will run 3--4 parameter
sweeps, this is clearly worthwhile. For a one-time experiment, it may not be.

Figure~\ref{fig:crossover} shows the crossover point at which caching becomes
cost-effective.

Global feature selection (used in this thesis) introduces minor label leakage
but enables a stable fingerprint across all folds. The leakage mechanism is
structural: with 128 subjects, each LOSO fold holds out 1 subject whose labels
nonetheless contributed to the global feature ranking, representing
$1/128 \approx 0.8\%$ of the selection data. An empirical benchmark
(Figure~\ref{fig:global_vs_perfold}) measured this effect directly: the
mean accuracy difference across all configurations and subject counts is
$\pm 0.4\%$ with a maximum of $1.1\%$, and the direction is inconsistent
(per-fold occasionally outperforms global, confirming the effect is noise not
bias). The warm-run speedup benefit of global scope is more significant: XGBoost
retains its full 14--23$\times$ warm speedup, whereas per-fold scope degrades to
4--9$\times$ because feature selection re-runs for every fold even on cache
hits. Global selection is therefore the pragmatic choice for caching-oriented
workflows.

\subsection{Limitations}

\begin{itemize}
\item \textbf{Single dataset:} All experiments were conducted on the BOAS
      dataset (128 subjects). Generalisation of cache viability verdicts to
      other EEG datasets or non-EEG domains requires further validation.

\item \textbf{Global feature selection:} Feature selection is performed across
      all subjects before LOSO splits, introducing minor label leakage
      ($1/128 \approx 0.8\%$ of selection data per fold). An empirical benchmark
      showed the resulting accuracy difference is $\leq 1.1\%$ with no
      consistent direction (Section~\ref{sec:experiment_setup}). A strictly
      correct per-fold implementation would eliminate this leakage at the cost
      of reduced warm-run cache speedup.

\item \textbf{No deep learning comparison:} The framework was evaluated only
      on traditional ML models. Deep learning models (e.g., U-Sleep, CNNs)
      have different serialization characteristics and may exhibit different
      cache viability patterns.

\item \textbf{No multi-user evaluation:} The framework was designed for
      single-researcher workflows. Cache sharing across multiple concurrent
      users would require locking mechanisms not implemented here.
\end{itemize}
